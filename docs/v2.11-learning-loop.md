# v2.11 — Learning loop design (locked)

Status: design locked 2026-05-25 via war-room `0B0685A2-949D-4AFE-ABF2-0FAEC084C05A` (2 seats: claude-sonnet-4-6 CLI + gemini-2.5-pro API). Both reviewers converged independently on every question. Implementation tracks v2.11 PR-12.

## What we're building

The YC self-improving-company framework has a learning-mechanism step: failure → diagnose → propose change → review → deploy. v2.10 ships the failure-detection half (methodology runner + regression-watch). v2.11 ships the rest, scoped to **agent-definition changes** — because the AI itself doesn't learn (model weights are frozen); only the system around it does.

Concretely: a failing cell in a `methodology_run` → a "diagnose" agent proposes a specific change to the agent's definition file → that change applies to a **variant copy** of the agent → the same methodology re-runs against the variant → ship the variant only if it Pareto-improves with statistical significance (and doesn't overfit).

## Locked decisions

### Q1 — Input shape: tiered, curated subset

NOT "all 157 dispatches" (50K+ tokens) and NOT "just the agent definition" (zero signal). Send three sections in one prompt, target ~30K-token envelope:

1. **Per-cell aggregate stats** — one row per (prompt_idx, model, condition): mean score, mean cost, n, mean tokens_out, Welch p vs. the best-scoring cell. ~50-200 rows / ~2K tokens regardless of run size.
2. **The current agent definition file** verbatim (`~/.claude/agents/<slug>.md` or the runtime-specific equivalent). ~2-5K typical.
3. **Worst-K + best-K dispatches verbatim** (prompt + response + score + rubric reason), sampled from the lowest-scoring and highest-scoring cells. Default K=3 worst-cell / K=2 best-cell, hard cap 30 dispatches total. ~15-25K.

**Best-K is non-negotiable**: without positive anchors the diagnose agent rewrites away from things that already work.

`--full-traces` flag overrides the cap for obstinate cases.

### Q2 — Output shape: restrictive structured JSON

NOT raw diff (LLM-generated diffs against prose `.md` files fail `git apply` ~20% of the time on whitespace + context). NOT a PR (PR creation happens AFTER the A/B step picks a winner, not before — otherwise reviewers drown in proposals that never made it through evals).

Schema:

```json
{
  "variant_slug": "code-reviewer-v2",
  "rationale": "Agent is failing on complex refactoring tasks because system prompt over-emphasizes caution. Loosening the 'never delete' rule.",
  "changes": [
    {
      "target_file": "~/.claude/agents/code-reviewer.md",
      "operation": "replace_section",
      "section_marker": "## System Prompt",
      "content": "..."
    }
  ],
  "expected_improvements": [
    {"prompt_idx": 16, "condition": "advisory", "predicted_delta": "+0.20"}
  ]
}
```

Operations enum is intentionally narrow: `replace_section | append | prepend | replace_file` (claude) or `REPLACE_BLOCK | APPEND_TO_LIST | INSERT_AFTER_LINE` (gemini). Pick the **claude variant** for v2.11 — it's the right level of expressiveness for markdown agent definitions (most ATO agents are `.md`, not `.yaml`).

If the diagnose agent can't express its change in those four operations, the change is too clever and should be rejected. That restrictiveness is the point.

### Q3 — Diagnose model: claude-opus-4-7 default, override-able

Diagnose is meta-cognition (reasoning about why an agent failed and proposing prompt changes). Opus is best at this workload. The dollar delta vs. sonnet-4-6 on a single 30K-token diagnose call is ~$0.40; the A/B run that follows costs $5-50; cheaping out on diagnose to save 1% of total cost is false economy.

Expose two override surfaces:
- `--diagnose-model <id>` CLI flag (per-invocation override)
- `diagnose_model` field in the methodology config JSON (methodology-level default)

Default fallback chain: `claude-opus-4-7` → `claude-sonnet-4-6` → `gemini-2.5-pro` (per CLAUDE.md auth-resolution order).

**Deferred to v2.12**: run diagnose itself as a methodology cell (N proposals from N models, pick the one whose A/B wins). Worth flagging here so the field name doesn't become a one-model assumption.

### Q4 — A/B mechanics + win condition

**Mechanics:**

- Variant slug: `<original>-v<N>` where N auto-increments per (original, methodology) pair.
- New `methodology_run` row inserted with a new schema column: `parent_run_id` TEXT (nullable, FK semantics not enforced). The variant run uses the SAME `methodology_id` so its `variant_matrix` is byte-identical — same prompts, same models, same conditions, same reps.
- The variant agent definition file lives at `~/.claude/agents/<original>-v<N>.md` (or the runtime-equivalent path per CLAUDE.md).

**Win condition** — "Statistically Significant Pareto Improvement" (both reviewers converged):

1. **≥1 cell shows mean-score improvement with Welch t p<0.05** (v2.10 PR-9 ships this).
2. **NO cell shows mean-score regression with Welch t p<0.05.**
3. **Mean cost per cell does not increase by >10% in any cell**, UNLESS that cell also hit condition (1) with a delta ≥0.2.

Why not strict Pareto: with reps=30 and natural LLM variance, *some* cell will appear worse by chance. Strict Pareto blocks every variant on noise.

Why not "mean improved": that lets you ship a variant that improved one prompt by 0.4 while silently destroying another by 0.3.

Decision lands in `apps/cli/src/methodology/diagnose.rs::compare_runs` with three predicates that read like the rule:
- `any_significant_improvement(baseline, variant) -> bool`
- `any_significant_regression(baseline, variant) -> bool`
- `cost_inflation_unjustified(baseline, variant) -> bool`

### Q5 — Surface

**CLI** (canonical):
```
ato evaluations methodology diagnose <run-id>
    [--diagnose-model <id>]
    [--apply]    # write the variant file to disk
    [--ab]       # also run the variant methodology
    [--pr]       # gh pr create the variant (after --ab confirms win)
```

Lives in `apps/cli/src/commands/methodology.rs`. Fits the existing mental model: `evaluations → methodology → {create, list, get, run, cost-estimate, schedule, calibrate, diagnose}`.

**NOT a new `ato learning-loop` top-level** — only one loop step today. Top-level namespace deserves to wait until Langfuse ingestion + propose-from-traces + multi-step adaptive iteration land (v2.12 or v2.13). Premature top-level namespaces are sticky; the `evaluations methodology diagnose → learning-loop diagnose` rename is cheap.

**MCP tool**: yes. `diagnose_methodology_run` in `services/mcp-server/src/tools/methodology.ts`. This is the high-leverage surface — an AI driving ATO end-to-end (create methodology → run → diagnose → A/B → ship) is the YC self-improving-company demo. CLI is foundation; MCP is multiplier.

**Desktop surface**: a "Propose improvement" button on the methodology-run detail view in `MethodologiesPanel.tsx`. Same screen, new action. No new view needed.

### Q6 — Langfuse signal (Mode A ingestion)

Same call, `production_signal:` structured block in the diagnose prompt. NOT a separate pass — synthesis between "dev eval says X" and "prod says Y" is the highest-value reasoning step, and pushing it to the runner means doing it badly with hardcoded weights.

Diagnose system prompt gets a 3-line addition: **"When dev evals and production signals conflict, production wins; rubric scores measure what we thought to ask, production measures what users actually experience."**

Storage: new OSS table `production_signals (id, agent_slug, source, signal_json, captured_at)` in `schema.rs` so the Langfuse ingester (closed-source per CLAUDE.md, lives in `ato-cloud`) has a write target the OSS diagnose pipeline can read. Clean OSS/cloud boundary.

Ingest command lands in a follow-up PR — v2.11 ships the schema + the diagnose pipeline that consumes it; the ingester itself sits behind the cloud feature flag.

### Q7 — The load-bearing risk (both reviewers flagged the same thing)

**Methodology overfitting / Goodhart's law.** The diagnose agent gets the failing prompts → proposes a change that fixes those prompts → A/B re-runs the same methodology → variant wins → you ship → prod users encounter a prompt the methodology never covered → variant is worse than the original on that case, possibly catastrophically.

This is the central pathology of every learning loop. We bake three defenses into v2.11 **before shipping**:

#### 1. Holdout set per methodology (REQUIRED)

Add a `holdout_prompts: Vec<String>` field to `VariantMatrix` in `apps/cli/src/methodology/types.rs`. The diagnose agent NEVER sees these prompts (they're not in the per-cell stats it reads, not in the worst-K/best-K sampling). The A/B win condition (Q4) must hold on holdout cells too, not just visible cells. **If holdout regresses, variant is rejected even if visible cells improve.**

This single rule kills ~80% of the overfitting risk.

#### 2. Variant lineage tracker

New table `agent_variant_lineage (variant_slug, parent_slug, generation, created_at, run_id_that_birthed_it)`. When the chain reaches depth ≥3 within 14 days, the diagnose CLI prints a warning: *"this agent has been auto-modified 3 times in 14 days; consider a manual review."* Compounding small wrong moves is the failure mode the depth counter catches.

#### 3. Auto-revert window (post-ship Langfuse watch)

Every shipped variant gets a 7-day Langfuse watch period. If prod metrics regress vs. the previous variant by >2σ on any tracked metric, the variant is auto-reverted to its parent and the diagnose pass is replayed with the regression evidence appended.

This is the loop closing on itself. Without it, v2.11 is an open-loop drift machine labeled as a learning loop.

(Implementation depends on the Langfuse ingester from §Q6; auto-revert lands as a v2.11.x follow-up. The 7-day window is the design contract; the implementation can stage in two PRs.)

## Schema deltas (locked)

Three additive schema changes for v2.11. All non-destructive.

```sql
-- v2.11 PR-12.0 — Variant lineage (Q7 defense #2)
CREATE TABLE IF NOT EXISTS agent_variant_lineage (
    variant_slug   TEXT PRIMARY KEY,
    parent_slug    TEXT NOT NULL,
    generation     INTEGER NOT NULL DEFAULT 1,
    created_at     TEXT NOT NULL,
    birthed_by_run TEXT NOT NULL,
    diagnose_model TEXT
);
CREATE INDEX idx_variant_lineage_parent ON agent_variant_lineage(parent_slug, generation);

-- v2.11 PR-12.0 — A/B linkage (Q4)
ALTER TABLE methodology_runs ADD COLUMN parent_run_id TEXT;
CREATE INDEX idx_methodology_runs_parent ON methodology_runs(parent_run_id);

-- v2.11 PR-12.0 — Holdout prompts (Q7 defense #1)
-- variant_matrix already a JSON blob; we extend its shape, no DDL needed.

-- v2.11 PR-12.x — Production-signals ingestion target (Q6, schema-only in v2.11)
CREATE TABLE IF NOT EXISTS production_signals (
    id           TEXT PRIMARY KEY,
    agent_slug   TEXT NOT NULL,
    source       TEXT NOT NULL,    -- 'langfuse' | 'helicone' | 'manual'
    signal_json  TEXT NOT NULL,
    captured_at  TEXT NOT NULL
);
CREATE INDEX idx_production_signals_agent ON production_signals(agent_slug, captured_at DESC);
```

## Open questions still on Will's plate

These the war-room couldn't answer; they need product judgment:

1. **Auto-vs-manual gate on `--apply`.** Should `--apply` require a `--yes` confirmation by default, or just write the variant file and let the maintainer review the diff? Default-on `--yes` ships faster; default-off respects "ATO doesn't change your files behind your back" doctrine. Recommended: default-off (the YC talk says "human in supervision capacity").
2. **Variant retention.** When a variant loses the A/B, do we delete its file or keep it for forensics? Recommended: keep, archive under `~/.claude/agents/.archive/<slug>-<run-id>.md` so the diff trail survives.
3. **Pricing tier.** Diagnose pass is meta-LLM-call ($0.40-$1 per call). Free or Pro? Both reviewers stayed silent here. Recommended: free (the user pays their own API), but `--ab` execution counts against the customer's methodology spend budget if they've set one.
