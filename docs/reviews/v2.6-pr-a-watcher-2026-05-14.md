1. **MEDIUM — `PassiveObserver::start()` not idempotent on `dirs::home_dir()` failure**
   - **Description**: When `dirs::home_dir()` returns `None`, `start()` returns `Err(...)` at line 73 without setting `self.started = true`. The `started = true` assignment only executes at line 134, after the watcher thread is spawned. This means a second call to `start()` on the same instance will pass the `if self.started { return Ok(()); }` guard and retry `dirs::home_dir()`, hitting the same error again. The Tauri setup hook in `lib.rs` calls `start()` once and eprintfs any error, so in normal launches this is silent. But if the setup hook fires twice (e.g., a future hot-reload iteration), the second invocation will retry and re-eprintf the same error every frame tick.
   - **Location**: `apps/desktop/src-tauri/src/passive_observer.rs:73-74`
   - **Fix**: Set `self.started = true` before the `dirs::home_dir()` check, or set it in the early-return arm:
     ```rust
     None => {
         self.started = true; // <-- add this
         return Err("home directory unknown".to_string());
     }
     ```
   - **Verified via**: `read_file passive_observer.rs:68-80` — confirmed `started = true` only at line 134, after the home-dir branch.

2. **MEDIUM — Passive observations always attribute to `_subscription` billing surface; cannot distinguish API-key users**
   - **Description**: Every passive observation row sets `billing_surface` to the subscription default (`claude_code_subscription` / `codex_cli_subscription`). The comment acknowledges this as a known honest default and defers correction to PR-B (cloud polling). However, the UI immediately surfaces these rows in the "Last 7 days at a glance" card and the billing-surface group-by, so a user running Claude Code on `ANTHROPIC_API_KEY` will see their observed turns counted as "subscription hours" and the `SurfaceRow` will show "flat-rate" instead of "per-token." The `auth_mode` column is also `NULL` for passive rows since `effective_auth_mode_from_path` can't read the upstream process's env vars.
   - **Location**: `apps/desktop/src-tauri/src/passive_observer.rs:840-841` (`emit_row` sets `billing_surface: Some(kind.default_billing_surface())`), and the same in `mark_passive_in_progress` at line 882.
   - **Fix**: Document the limitation prominently in the UI beside every billing-surface row (the blind-spot caveat already exists on the glance card but not on individual `SurfaceRow` entries). Alternatively, emit a `billing_surface: None` for passive rows until PR-B lands, so they appear as "unknown" rather than mis-attributed.
   - **Verified via**: `read_file passive_observer.rs:840-841` — confirmed `default_billing_surface()` is the only source of billing surface for passive rows.

3. **LOW — Codex `token_count` events are latched to all sessions, not scoped to the current pair**
   - **Description**: The Codex `event_msg` handler iterates over `state.sessions.values_mut()` to set `pending_tokens_in/out`. The comment explains this is because "there's only ever one session per rollout file." This is correct for today's Codex format but is an unnecessarily global mutation — if a future Codex version emits multi-session files, or if the `SessionStateMap` were reused across files, all sessions would receive the same token counts. For single-session files the behavior is correct by coincidence.
   - **Location**: `apps/desktop/src-tauri/src/passive_observer.rs:693-699`
   - **Fix**: Scope the token latch to the single known session. Since `session_meta` is the first line of each Codex rollout file and establishes the session ID, the map will have exactly one entry at the time `event_msg` arrives:
     ```rust
     if let Some(pair) = state.sessions.values_mut().next() {
         pair.pending_tokens_in = t_in;
         pair.pending_tokens_out = t_out;
     }
     ```
   - **Verified via**: `read_file passive_observer.rs:690-699` — confirmed loop over `state.sessions`.

4. **LOW — Codex `token_count` events may arrive out of order, applying counts to the wrong turn**
   - **Description**: If a Codex `token_count` event fires after a user message but before the assistant response (or after the assistant but before the next user), the latched counts will be attributed to the next `emit_row` call rather than the one that actually incurred those tokens. Without real Codex JSONL samples it's impossible to confirm the event ordering, but the risk is real for any streaming API where token events can trail the message that triggered them.
   - **Location**: `apps/desktop/src-tauri/src/passive_observer.rs:681-699`
   - **Fix**: Add a comment acknowledging the ordering assumption, and consider tagging token events with a turn sequence number if Codex emits one, so tokens can be matched to the correct pair.
   - **Verified via**: `read_file passive_observer.rs:681-699` — the `token_count` latch has no ordering validation.

5. **INFO — `worker_loop` receives `sources` but never uses it**

The initial scan sends all file paths via the `tx` channel before the worker starts, so the worker's knowledge of sources is only needed for type dispatch — which it gets from `ScanRequest::kind` instead.
   - **Location**: `apps/desktop/src-tauri/src/passive_observer.rs:248`
   - **Fix**: Remove the parameter: `fn worker_loop(db_path: PathBuf, rx: Receiver<ScanRequest>)`. Or rename `_sources` → `(_sources: Vec<Source>)` with the outer parens to make the intentional discard explicit.
   - **Verified via**: `read_file passive_observer.rs:246-320` — `_sources` appears only in the signature, not in the body.

**1. My own findings I'd withdraw or downgrade:**

- **Finding #1 (MEDIUM — `start()` not idempotent on `dirs::home_dir()` failure)**: I'd downgrade to **LOW / INFO**. After re-reading `lib.rs:1420-1429`, `start()` is called exactly once from the `setup` hook. There is no evidence in this diff of any multi-call path. The idempotency guard at the method entry (line 68: `if self.started { return Ok(()); }`) means a second call from the same setup hook on the same instance would be a no-op — the first call would set `started = true` even on failure if the home-dir check passed, but would return `Err` *without* setting it if `dirs::home_dir()` failed. However, the early `sources.is_empty()` arm correctly sets `started = true` before returning `Ok(())`, so the only non-idempotent path is the `dirs::home_dir() == None` branch. On a system where `HOME` is unset, this would produce a noisy eprintln on every boot attempt after the first — annoying but not data-corrupting. Reviewer 1's fix is still correct; I'm just lowering the severity.

- **Finding #3 (LOW — Codex token latch applies to all sessions)**: Withdrawn. Codex `rollout-*.jsonl` files are confirmed 1-session-per-file, and the `process_codex_line` session-lookup at line 730 (`state.sessions.keys().next()`) is also predicated on that invariant. The loop is harmless. My own Finding #4 (ordering) is the real concern; Finding #3 was a false alarm from over-applying the "don't mutate shared state" heuristic.

**2. Findings I actively disagree with from the other reviewer:**

- **Finding #2 from Reviewer 1 (MEDIUM — "misleading `_subscription` attribution for API-key users")**: I **disagree with the framing as a bug worth blocking merge**. The PR explicitly documents this as v2.6's honest stance: "we cannot peek inside the upstream CLI's process to read its env vars." The UI already communicates this via the blind-spot caveat on the glance card. A user paying per-token for Claude Code will see "subscription hours" which is wrong in absolute terms, but the alternative (emitting `unknown`) is strictly worse — it loses all observability for the majority of users who *are* on subscriptions. Reviewer 1's "Fix: emit `billing_surface: None` for passive rows" would result in `unknown` for every passive observation, defeating the PR's entire observatory positioning. The right call is to keep the default and let PR-B fix it when cloud-side token reconciliation lands. Not a merge blocker.

- **Finding #1 from Reviewer 1 (MEDIUM — `start()` idempotency on `dirs::home_dir()` failure)**: Partially disagree on severity. See above — the bug exists but only triggers on a pathological `HOME`-unset environment that most desktop apps don't bother defending against. The fix is trivial (add `self.started = true` before `return Err(...)`) and worth doing, but it shouldn't be MEDIUM since the failure is non-destructive (just an eprintln). Downgrade to LOW.
