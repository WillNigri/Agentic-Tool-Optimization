1. **HIGH — `reap_dead_live_runs` silently becomes a no-op on Windows**
   - Description: The zombie reaper uses `Command::new("kill")` which is a Unix-only command. On Windows, `kill` doesn't exist in the PATH, so spawning will fail. The code handles spawn failure by returning `false` ("conservative, leave row alone"), meaning the entire reaper silently degrades to a no-op on Windows. Every Windows user will see zombie `live_runs` rows indefinitely.
   - Location: `apps/desktop/src-tauri/src/active_runs.rs:464`
   - Fix: Wrap with `#[cfg(not(target_os = "windows"))]` and provide a Windows alternative (e.g., `tasklist` + parse, or a Windows-specific crate like `winapi`). On Windows, either skip the reaper or use `windows::Win32::Foundation::OpenProcess` with `PROCESS_QUERY_LIMITED_INFORMATION`.
   - Verified-via: `read_file apps/desktop/src-tauri/src/active_runs.rs:460-475`, `grep target_os.*windows health_poller.rs` (shows the pattern is used elsewhere in the crate but NOT in `active_runs.rs`)

2. **HIGH — `gemini` in `KNOWN_RUNTIMES` is seeded but never probed or filtered**
   - Description: `KNOWN_RUNTIMES` has 5 entries: `["claude", "codex", "gemini", "openclaw", "hermes"]`. The seed function creates a `runtime_preferences` row for `gemini`, and users can toggle it on/off via `setRuntimeMonitored`. However, `check_all_runtimes()` only iterates over `["claude", "codex", "hermes", "openclaw"]` — `gemini` is never checked. Toggling `gemini`'s monitor preference has zero effect; the row is dead UI.
   - Location: `apps/desktop/src-tauri/src/health_poller.rs:85` (`runtimes = vec!["claude", "codex", "hermes", "openclaw"]`) vs `apps/desktop/src-tauri/src/commands.rs:2918` (`KNOWN_RUNTIMES` includes `"gemini"`)
   - Fix: Either add `gemini` to `check_all_runtimes()` and implement `check_gemini_health()`, or remove `"gemini"` from `KNOWN_RUNTIMES` until it's implemented.
   - Verified-via: `read_file apps/desktop/src-tauri/src/health_poller.rs:84-86`, `read_file apps/desktop/src-tauri/src/commands.rs:2918`

3. **MEDIUM — `ensure_runtime_preferences_seeded` silently swallows all SQLite errors with `unwrap_or(0)`**
   - Description: The `query_row` call uses `.unwrap_or(0)` which converts any database error (locked, corrupt, permission) into "exists = false". Combined with the `if exists > 0 { continue }` check, every DB error causes the function to attempt an INSERT for every runtime on every call. This is both wrong (treating errors as "runtime not seeded") and potentially a performance issue (repeated INSERT attempts under DB contention).
   - Location: `apps/desktop/src-tauri/src/commands.rs:2929`
   - Fix: Propagate errors or at least distinguish "row doesn't exist" from "DB error":
     ```rust
     let exists: bool = conn
         .query_row(
             "SELECT 1 FROM runtime_preferences WHERE runtime = ?1",
             [runtime],
             |_| Ok(true),
         )
         .unwrap_or(false);
     ```
   - Verified-via: `read_file apps/desktop/src-tauri/src/commands.rs:2925-2937`

4. **MEDIUM — Grace window silently bypassed when `started_at` parses as invalid RFC3339**
   - Description: The grace-window check is `if let Ok(parsed) = chrono::DateTime::parse_from_rfc3339(&started_at)`. If the parse fails (malformed data, timezone edge case), the `continue` is skipped entirely and the row proceeds to `kill -0`. Rows <30s old with a corrupted `started_at` will be immediately reaped. This is the opposite of the intended safety behavior.
   - Location: `apps/desktop/src-tauri/src/active_runs.rs:453-456`
   - Fix: Fail-safe: treat parse errors as "skip this row":
     ```rust
     let Some(parsed) = chrono::DateTime::parse_from_rfc3339(&started_at).ok() else { continue; };
     if (now - parsed.with_timezone(&chrono::Utc)).num_seconds() < 30 {
         continue;
     }
     ```
   - Verified-via: `read_file apps/desktop/src-tauri/src/active_runs.rs:453-456`

5. **LOW — `is_runtime_monitored` re-opens the DB on every call, defeating the WAL optimization goal**
   - Description: The inline query in `get_health_status` was specifically added to avoid a second `Connection::open` that "could contend in WAL mode." However, `is_runtime_monitored` (called from the health poller loop) opens a new connection, calls `ensure_runtime_preferences_seeded` (which can INSERT), then queries — all on a fresh connection. Under active health polling (every 30s × N runtimes), this creates repeated open/close cycles. The inline query in `get_health_status` is fine, but `is_runtime_monitored` is the call site that matters most.
   - Location: `apps/desktop/src-tauri/src/commands.rs:2986-2998`
   - Fix: Pass the existing `&Connection` from the poller's loop into `is_runtime_monitored`, or accept the WAL contention (SQLite WAL handles concurrent readers well; the concern is more theoretical than real here).
   - Verified-via: `read_file apps/desktop/src-tauri/src/commands.rs:2986-2998`, `read_file apps/desktop/src-tauri/src/health_poller.rs:93-96`

6. **LOW — `probe_cli_runtime` shell path has no quoting for paths with spaces**
   - Description: When `path.contains(' ')`, the code runs `sh -c "{} --version"` without quoting. A path like `/home/user/my tools/claude` would be parsed as three tokens (`/home/user/my`, `tools/claude`, `--version`) by the shell, breaking the probe. The code acknowledges this case but doesn't handle it safely.
   - Location: `apps/desktop/src-tauri/src/health_poller.rs:169-171`
   - Fix: Quote the path in the shell command:
     ```rust
     Command::new("sh")
         .args(["-c", &format!("\"{}\" --version", path)])
         .output()
     ```
     Or better: split the path on the first `/` or last `/` to get the directory + binary name, then use `Command::new` directly with `arg("--version")`.
   - Verified-via: `read_file apps/desktop/src-tauri/src/health_poller.rs:169-171`
