# v2.18 — Active Workstation (Web → Tethered Desktop)

> Promote the v2.17 tether channel from read-only event decryption to a
> **live workstation surface**: dispatch LLM runtimes from the browser,
> stream the reply back, and audit every call exactly like a local
> dispatch.

## 1. Why

v2.17 shipped the cryptographic + presence base: the browser can pair
with a running desktop over an E2E-encrypted DH session, and use it as
a remote oracle for read-only operations (`decrypt_events` for v2.15
team events).

The user goal — quoting Will, 2026-06-15: "get the web version in a
way we can use it to call and start working from there when the pc is
open like Claude does." The Claude Code reference here is the way
Claude.ai can drive a local terminal session: open browser, type a
prompt, get a response that came from the user's own machine + auth.

ATO already has the dispatch primitive (`commands::dispatch::run`),
the tether channel, and the audit pipeline (`execution_logs`). What's
missing is the glue: the browser-side UI, two new frame kinds on the
tether channel, and the desktop-side dispatch handler.

## 2. Out-of-scope (deferred to v2.19+)

- Standalone web (no desktop tether) — that's v2.20, see project memo.
- Cross-machine dispatch routing (browser picks which paired host to
  run on) — Wave 4 if cheap, otherwise v2.19.
- Editing files from the browser (multi-frame protocol, security model
  is materially heavier).
- Persistent terminal sessions (xterm-over-tether) — v2.20.
- Subagent observability surface in the browser — read-only via
  shared workspace renderers (v2.16) already shows these.

## 3. Frame schema

The tether channel already routes one app-frame kind (`decrypt_events`)
through the AEAD layer (X25519 → HKDF-SHA256 → XChaCha20-Poly1305).
v2.18 adds three more:

### 3.1 `dispatch_request` (browser → desktop)

```jsonc
{
  "kind": "dispatch_request",
  "request_id": "<uuid>",
  "runtime": "claude" | "codex" | "gemini" | "openclaw" | "hermes",
  "prompt": "...",
  "model": "claude-sonnet-4-6",          // optional
  "agent_slug": "review-bot",            // optional
  "war_room_id": "<uuid>",               // optional
  "war_room_round": 1,                   // optional
  "workspace_root": "<abs path>",        // optional; MUST be within
                                         // registered project roots
  "require_tools": ["read_file", "grep"],// optional (grounded mode)
  "session_id": "<uuid>"                 // optional (sticky session)
}
```

### 3.2 `dispatch_chunk` (desktop → browser, 0..N per request)

```jsonc
{
  "kind": "dispatch_chunk",
  "request_id": "<uuid>",
  "chunk_index": 0,
  "text": "..."
}
```

Streaming is best-effort. CLI runtimes that batch-flush (codex, claude
without --stream) will fire ONE chunk at the end. Browser MUST handle
chunk_index gaps gracefully — the relay never reorders, but the
desktop may buffer.

### 3.3 `dispatch_complete` (desktop → browser, exactly 1)

```jsonc
{
  "kind": "dispatch_complete",
  "request_id": "<uuid>",
  "status": "success" | "failed" | "denied" | "cancelled",
  "execution_log_id": "<uuid>",
  "cost_usd": 0.012,
  "tokens_in": 412,
  "tokens_out": 803,
  "model": "claude-sonnet-4-6",
  "duration_ms": 4123,
  "error": null                  // populated when status != success
}
```

### 3.4 `dispatch_cancel` (browser → desktop, optional)

```jsonc
{
  "kind": "dispatch_cancel",
  "request_id": "<uuid>"
}
```

If received before the runtime returns, the dispatch is aborted and
`dispatch_complete.status = "cancelled"` is sent. After completion,
cancel is a no-op (cannot un-spend the cost).

## 4. Security model

### 4.1 Trust boundary

The tether channel is already authenticated:

- **Cloud relay** is untrusted (assumes pull-only routing; never sees
  plaintext).
- **Browser-host pair** is mutually authenticated via X25519 DH +
  Ed25519-signed handshake. Per-machine `browser_pubkey_fp` displayed
  to the user during pairing for shoulder-surfer resistance.
- **AEAD session key** has 96-bit nonce derived from
  `(direction, seq)` — replay-protected.

`dispatch_request` rides this same channel, so the wire-level threat
model is unchanged.

### 4.2 Authorization (the new question)

**Pair-level vs per-dispatch approval.** v2.17 has two approval modes:

- `once` — one session, then re-prompt on next pair.
- `always` — persistent row in `tether_approvals` for this device.

v2.18 dispatch raises the stakes: a `dispatch_request` can spend
budget, write files (depending on tool grants), and reveal source
code via tool calls. Options:

**Option A — pair-level only (recommended for Wave 1)**: rely on the
`always` approval. Pros: matches Claude.ai UX; user already
consented to this browser-machine pair for the workspace. Cons:
shoulder-surfer with a paired browser can fire dispatches with no
extra prompt.

**Option B — per-dispatch confirmation**: every `dispatch_request`
raises a desktop modal. Pros: maximal safety. Cons: defeats the
"feels native" UX, makes the feature nearly unusable for power users
who dispatch dozens of times per session.

**Option C — risk-tiered**: free runtimes (subscription-billed) pass
without prompt; budget-spending API calls (anthropic_api,
openai_api, etc.) prompt only above a per-day threshold. Pros:
matches the harm-cost gradient. Cons: more code, more state, more
edge cases.

**Wave 1 ships A**. Wave 2+ can layer C if dogfooding shows the gap.

### 4.3 Workspace_root scoping

The browser MAY send `workspace_root` to set the CWD for the
dispatch. Desktop MUST validate that the path falls under one of the
user's registered project roots (read from `projects` table in the
local DB). Out-of-bounds → reject with `dispatch_complete.error =
"workspace_root not registered"`.

### 4.4 Tool grants

`require_tools` is a tightening directive only (matches `ato dispatch
--require-tools`). The browser cannot grant additional capabilities;
the desktop's agent record + machine policy is the floor. This is
already enforced by the existing dispatch pipeline.

### 4.5 Audit row

Every browser-initiated dispatch writes an `execution_logs` row with:

| Column | Value |
|---|---|
| `initiator_kind` | `agent:browser` |
| `client_surface` | `tether_browser` |
| `initiator_id` | `<browser_pubkey_fp>` (12 hex chars) |
| `auth_mode` | resolved from runtime credential type |
| `billing_surface` | resolved from runtime credential type |
| `git_commit_sha` | captured via `dispatch::capture_git_head` (2s timeout, same as #78) |
| `war_room_id` | passed through if set |

The row is visible in the desktop Sessions feed and in the shared
workspace web view (v2.16). The audit trail is symmetric with local
CLI / Claude Code subagent dispatches (#71).

## 5. UX flow

### 5.1 Entry point

Shared workspace detail page (v2.16) already has a `TetherStatusCard`
that shows pairing state. Wave 1 adds a sibling card:
`WorkstationDispatchCard`. Visible only when tether status is
`approved` AND `host_online = true`.

### 5.2 Card contents

```
┌─ Run on <machine_name> ─────────────────────────────┐
│ Runtime: [claude ▾]   Model: [default ▾]            │
│ Prompt:                                              │
│ ┌──────────────────────────────────────────────────┐│
│ │                                                  ││
│ │                                                  ││
│ └──────────────────────────────────────────────────┘│
│ [Workspace: <project> ▾]      Tools: [□ require_tools]│
│                                                      │
│ [ Send ▶ ]                                           │
└──────────────────────────────────────────────────────┘
```

### 5.3 In-flight state

On Send, card swaps to:

```
┌─ Running on <machine_name> ─────────────  [Cancel ✕] ┐
│ Request <request_id_short>                            │
│ ┌────────────────────────────────────────────────────┐│
│ │ <streamed chunks render here>                      ││
│ │ █                                                  ││
│ └────────────────────────────────────────────────────┘│
└───────────────────────────────────────────────────────┘
```

### 5.4 Completion

On `dispatch_complete`:

```
┌─ Completed on <machine_name> ──────  [↑ View receipt] ┐
│ Status: success     Model: claude-sonnet-4-6           │
│ Tokens: 412 in / 803 out    Cost: $0.012   4.1s        │
│ ┌────────────────────────────────────────────────────┐│
│ │ <full response>                                    ││
│ └────────────────────────────────────────────────────┘│
└────────────────────────────────────────────────────────┘
```

`View receipt` deep-links to the shared workspace receipt for the
new execution_log row (already shipped in v2.16 FOLLOWUPS #1).

## 6. Open questions for the war-room

1. **Per-dispatch approval (§4.2)** — accept Wave 1 as Option A
   (pair-level), or do we need risk-tiered (Option C) from day one?
2. **Streaming chunk policy** — for CLI runtimes that batch-flush,
   should the desktop chunk the buffered reply into N pieces (better
   UX) or emit one final chunk (truthful, matches local)?
3. **Cancellation guarantees** — if cancel arrives while the runtime
   is mid-call, do we (a) kill the child process, (b) detach and let
   it finish locally without sending chunks, or (c) only honor cancel
   if pre-spawn?
4. **War-room flows from the browser** — Wave 1 supports a single
   dispatch_request per send. War-room (N runtimes share war_room_id)
   would need browser-side coordination of N parallel requests. Wave
   3 or push to v2.19?
5. **Workspace_root selector** — the browser may not know the user's
   project list. Option: tether host sends a `project_list` frame on
   pair-approve and browser caches it. Option: implicit
   `workspace_root = active project's root` (no picker).
6. **Subscription-runtime UX** — claude (subscription) reads
   `~/.claude/` for auth; that's already on the desktop. No web auth
   step needed. Confirm there's no cross-contamination if multiple
   browsers connect to the same host.
7. **Cost ceiling on browser-side** — should the browser enforce a
   `--budget` cap before firing? Or rely on the desktop's quota
   gates (v2.15.4 pause-and-wake)?
8. **Subagent observability symmetry** — when claude on desktop fires
   a subagent (Task tool), the subagent uses `ato subagent log`
   (#71). Does the browser-initiated dispatch also need to surface
   subagent rows in its UI, or is the post-completion shared
   workspace view enough?

## 7. Implementation waves

| Wave | Scope | Reviewers |
|---|---|---|
| 1 | Schema + Rust handler + browser card. ONE runtime (claude), no streaming, pair-level auth. | claude + codex R1, codex R2 |
| 2 | Streaming chunks (request_id-keyed) for CLI runtimes that support `--stream` (API providers); buffered chunks for the others. | codex + gemini R1 |
| 3 | Multi-runtime + war-room support (browser fires N parallel `dispatch_request`s with shared `war_room_id`). | claude + codex + gemini R1 (war-room about war-rooms) |
| 4 | UX polish + CSO security audit + a11y review + final R1/R2. | cso + codex + claude |

## 8. Non-goals reminder

- This is NOT the standalone web client. The host must be running on
  the user's machine and paired. v2.20 considers a hosted-only path.
- This is NOT a public API. The `dispatch_request` schema is internal
  to the tether channel; we do not document it for third-party use.
- This is NOT a job queue. Browser firing 1000 `dispatch_request`s is
  not a supported pattern; rate-limit per-pair (Wave 2 hardening).
