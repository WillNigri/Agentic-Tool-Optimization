# v2.0.0 — External Agents / Hosted Deployment

## Why

ATO today is a tool for the *developer* building agents. It runs on their laptop and dispatches to their own CLI / API key. The customer-facing flow ends there: if they want their agent to talk to *other people* — website visitors, Slack users, email triage — they have to hand-roll the deployment themselves.

v2.0 turns ATO into the place where companies design customer-facing agents, deploy them to **their own infrastructure** (any of the 15 supported providers), and track behavior — **without ATO ever owning the inference compute**. We're "Stripe for agents," not "Vercel for agents." Customer keeps their Anthropic / OpenAI / Gemini relationship; we provide the IDE, the deployment templates, the embed widget, and the multi-runtime observability nobody else can do.

## Strategic position

| What we DON'T do | What we DO own |
|---|---|
| Run GPUs / inference compute | The agent IDE — variables, hooks, prompt versioning, evals |
| Compete with Cloudflare / Vercel / AWS | Deployment templates that target each of those |
| Hold customer conversation data unless they opt in | Static CDN for the embed widget |
| Compete with Helicone / Langfuse on request-level traces | Multi-runtime regression + agent-level observability |

**The provider is always the customer's choice.** All 15 supported providers (Claude, OpenAI, Gemini, Mistral, Groq, xAI/Grok, Together, Fireworks, DeepSeek, Qwen, MiniMax, Kimi, GLM, Yi, plus OpenClaw + Hermes for self-hosted setups) get equal treatment.

## Goal

A customer can:

1. Open ATO, build `acme-support` for their website
2. Mark it "External" → ATO locks it down (read-only, no shell, no fs writes)
3. Upload `policies.pdf`, `faq.md`, `product-spec.md` as knowledge (RAG)
4. Connect `https://acme.com/api/orders` as a scoped tool
5. Click Deploy → pick "Cloudflare Worker, Anthropic Claude API"
6. Get a `worker.js` they `wrangler deploy` to *their* account
7. Get an embed snippet they paste into `acme.com`
8. (Optional) Toggle "Send traces to ato-cloud" or "Stream to Langfuse"
9. Open Insights → see conversations / escalations / costs

End-to-end on the customer's infra, with us providing tooling at each layer.

## Scope

### In scope (v2.0.0)

- **Internal-vs-External toggle on agent create**. External unlocks Deploy + Knowledge tabs, locks permissions to read-only by default.
- **Knowledge ingestion**: file upload (PDF / MD / TXT / URL), chunking, embedding via the customer's chosen provider (Anthropic / OpenAI / etc.) or local embedding model. Stored in pgvector on `ato-cloud` (Pro+).
- **External APIs + DB connections** as scoped tools — already partially there via `agent_hooks` (db-query / mcp-call resolvers). Add a UI for tool registration.
- **Deploy targets** — code generators for the 4 main targets:
  - Cloudflare Worker (`worker.js`)
  - Vercel Edge Function (`api/agent.ts`)
  - Docker image (`Dockerfile` + `index.js`)
  - Standalone Node script (for self-hosted / Railway)
  - Each generator outputs a complete bundle: agent definition, system prompt, variables resolved at deploy time (or at runtime via env vars), context hooks, RAG retrieval, tool dispatch, eval scoring, optional trace POST-back.
- **Embed widget** — static JS hosted on `embed.agentictool.ai/v1.js`. Drops a chat bubble on any site. Talks to the customer's deployed endpoint (their Cloudflare Worker etc.). Tracks: conversation_started, message_sent, escalation_triggered, conversation_ended.
- **Provider-agnostic dispatch** in the deploy bundle. The customer picks any of the 15 providers. The bundle uses an OpenAI-compatible client where possible; falls back to provider SDK where required.
- **Tracking dashboard (Pro)** — Insights → External Agents tab. Per-deployed-agent: conversations/day, message volume, escalation rate, average conversation length, cost-per-conversation (estimated from the customer's reported usage).
- **Trace sink integrations** — one-click forward to Langfuse / Helicone / LangSmith / OpenTelemetry endpoint. We don't compete on request-level traces; we forward.

### Out of scope (deferred to v2.1+)

- Multi-runtime regression detection, cost optimization recs, eval workbench → v2.1.
- Real-time collaborative editing, team workspaces → v3.0.
- A/B routing, canary deploys, agent versioning UI → v3.0.
- Federated agent-to-agent calls → v4.0.

## Architecture

```
┌──────────────────────────┐         ┌─────────────────────┐
│ ATO Desktop (free OSS)   │  build  │ Customer's infra    │
│ - design agent           │  ────►  │ - Cloudflare Worker │
│ - upload knowledge       │ deploy  │ - Vercel Edge       │
│ - test locally           │  ────►  │ - Docker / Railway  │
│ - generate deploy bundle │         │ - their Anthropic / │
└──────────────────────────┘         │   OpenAI / etc. key │
            │                        └────────┬────────────┘
            │ optional                        │
            │ trace POST                      │
            ▼                                 ▼
┌──────────────────────────┐         ┌─────────────────────┐
│ ato-cloud (Pro)          │  embed  │ Customer's website  │
│ - knowledge / pgvector   │  ────►  │ <script src="embed  │
│ - agent_traces           │  serve  │   .agentictool.ai"> │
│ - Insights dashboard     │         │   end-user chat     │
│ - trace forwarder        │         └─────────────────────┘
└──────────────────────────┘
```

Three completely separate runtimes:
1. **Desktop** — IDE for designing agents. No inference. Local-first.
2. **Customer's infra** — runs the deployed agent. Talks to their LLM provider directly. Optionally posts traces back to ato-cloud.
3. **ato-cloud** — knowledge storage, traces, dashboard, embed widget CDN.

## Critical files (preview)

### Desktop

- `apps/desktop/src/components/AgentDetail/DeployTab.tsx` (new) — target picker, generated code preview, copy-to-clipboard, "open in browser" links to deployment provider.
- `apps/desktop/src/components/AgentDetail/KnowledgeTab.tsx` (new) — file upload, URL ingestion, chunk preview.
- `apps/desktop/src/components/AgentDetail/ExternalToggle.tsx` (new) — "Internal vs External" toggle on the create form + agent detail header.
- `apps/desktop/src/lib/deployBundleGenerators/{cloudflare,vercel,docker,node}.ts` (new) — code emitters for each target. Each takes an `Agent` record + the customer's deploy-time config and emits a complete deployable file.

### Cloud (ato-cloud)

- `services/embed/` (new microservice) — serves the embed widget JS, proxies trace POSTs to the agent_traces table.
- `services/knowledge/` (new microservice) — file upload, chunking via `langchain` / `llamaindex`, pgvector storage, retrieval API the deploy bundles call.
- `database/migrations/010_v2_external_agents.sql` (new) — `external_agents`, `agent_knowledge_chunks`, `agent_deployments`, `embed_conversations` tables.
- `services/skills/src/routes/agentTraces.ts` (extend) — accept traces from deployed embeds (rate-limited via embed key, not user JWT).

## Build order

**Wave 1 — agent shape (week 1-2)**
1. Internal-vs-External toggle + permissions auto-lock for external agents.
2. Deploy tab UI skeleton — picks target, shows placeholder code.
3. Cloudflare Worker code generator (smallest, ships first).

**Wave 2 — knowledge (week 3-4)**
4. KnowledgeTab UI — file upload, URL ingestion, chunk preview.
5. Cloud knowledge service — `POST /knowledge`, `POST /retrieve`. pgvector + embeddings via configurable provider (default OpenAI text-embedding-3-small).
6. Generated deploy bundles call `POST /retrieve` for RAG context.

**Wave 3 — deploy targets (week 5)**
7. Vercel + Docker + Node generators (port the Cloudflare logic).
8. Per-target test harness — generate, deploy locally to a stub server, send a message, verify response.

**Wave 4 — embed (week 6)**
9. Embed widget JS bundle — bubble UI, message stream, history.
10. Embed CDN service.
11. Trace POST-back from embed → cloud `agent_traces`.

**Wave 5 — observability (week 7)**
12. Insights → External Agents dashboard. Conversations / escalation / cost. Read from `agent_traces` + `embed_conversations`.
13. Trace sink integrations — Langfuse / Helicone / LangSmith / OpenTelemetry. Toggle in deploy bundle generator.

**Wave 6 — polish + docs (week 8)**
14. Docs: `docs/EXTERNAL-AGENTS.md` user guide, code examples per target.
15. Marketing site — landing page for "external agents" feature.
16. v2.0.0-IMPLEMENTATION.md, ROADMAP, memory updates.

Total: ~8 weeks. The unique-value observability features (regression detection, cost optimization, pipeline visualizer) push to v2.1.

## Verification checklist

When v2.0.0 ships:

1. Toggle an agent to External. Permissions auto-lock to read-only — confirm no shell/fs/write capabilities.
2. Upload a 10-page PDF as knowledge. Open Knowledge tab → see chunks, see embedding model used.
3. Test agent locally with knowledge enabled — confirm it cites the right chunk.
4. Click Deploy → Cloudflare Worker. Get the generated `worker.js`. Deploy with `wrangler` to a test account.
5. Embed the snippet into a static `index.html`. Open in browser — chat works end-to-end.
6. Send 5 messages from the embed → confirm they show up in Insights → External Agents dashboard.
7. Toggle "Stream traces to Langfuse" with a real Langfuse key — confirm traces arrive in Langfuse too.
8. Verify the conversation ran on the *customer's* Anthropic API key (check their Anthropic dashboard for usage).
9. Repeat (4)-(8) for the other 3 deploy targets (Vercel, Docker, Node).
10. Repeat (4) with each of the 15 providers configured — confirm the generated bundle works for each.
11. ROADMAP / PRD / V2.0.0-IMPLEMENTATION.md / CLAUDE.md updated.

## Open questions to lock before kickoff

1. **Embedding provider for knowledge**: default to OpenAI's text-embedding-3-small ($0.02/1M tokens) — but this means knowledge upload requires an OpenAI key even if the agent runs on Claude. Alternatives: local embedding via Ollama (slower but no key needed), or per-customer choice. Default + override?
2. **Embed widget styling**: how customizable? Branded by default, or always allow customer to override CSS / use a headless mode?
3. **Embed widget hosting cost**: at what scale does the CDN start hurting? Build budgets into pricing decision.
4. **Authentication for the embed**: how do we prevent anyone from spamming `acme-support`? Per-deploy embed key with rate limit + origin allowlist (cloudflare validates `data-key-ref` matches the request origin).
5. **Free tier deploy quota**: 1 deployed agent + 1k conv/mo, or unlimited deploys but throttled traces? User said keep open for now; we'll set the limit when we add billing.
