# v1.6.0 — Automations tab repurpose

## Why

The Runs → Automations tab today visualizes flow graphs auto-derived from `## Step N` / `## Phase N` headers inside SKILL.md files. It's a useful but narrow surface — most users don't have multi-step skills, so the tab usually shows one or two examples (`/skill-creator`, `/gstack`) and reads as filler.

v1.5 introduced two concepts that genuinely belong in a flow visualization:

1. **Sequential groups** — "Claude writes → Codex reviews → Gemini summarizes" pipelines where one prompt fires N stages, each on its own runtime. Today the Group editor has a Graph view but it's hidden inside `Agents → Groups → [group]`, and only shows the static topology — not what's running, not where it failed, not how long each stage took.
2. **Routed groups + cron jobs** — both run "automatically." They each have their own UI surface (Groups Form view, Runs → Schedules) but no shared place to see them as one operational picture.

The Automations tab is the natural home for all of this. It already has the canvas, the node primitives, and the execution overlay. v1.6 makes it earn its name by surfacing **everything that runs without a human in the loop** on one visualization, with live status.

## Goal

When a user opens **Runs → Automations** they see — at a glance — every automated thing in their workspace:

- **Sequential groups** as left-to-right flows with the writer / reviewer / summarizer nodes color-coded by runtime, arrows showing data flow, and a "stage X/N" pill on each.
- **Routed groups** as fan-outs: the router node in the middle, child agents radiating out, with each routing rule rendered as a labeled edge.
- **Cron jobs** as flow heads anchored on the left edge with a clock icon and the next-run timestamp; the rest of the flow is the agent / group the cron dispatches to.
- **Hooks** (pre-call context resolvers) attached to their parent agent as small input-side nodes.
- **Skill flows** — the existing skill-derived workflows — kept around as a sub-mode for users who already rely on them.

Each node carries live state: idle / running / succeeded / errored, with the last-run timestamp and a click-through to that agent's trace explorer.

## Scope

### In scope (v1.6.0)
- Source switch: tab loads three sources in parallel — agent groups (routed + sequential), cron jobs, and skill-derived flows. Merged into a single canvas.
- Live status per node, sourced from `~/.ato/agent-logs.jsonl` (last-run-by-agent index) — same file the Insights observability dashboard reads.
- Click-through: clicking any node opens that agent / group's Insights row pre-filtered to the last 50 runs.
- Filter chips: "Groups · Schedules · Skills · Hooks" — toggle visibility per source.
- Empty state copy that points users at the four ways automations are created (group, cron, skill, hook).

### Out of scope (deferred)
- Drag-and-drop authoring of new groups from the canvas (today groups are authored in the Groups Form view; v1.7+ could bring authoring here).
- Real-time WebSocket status updates — v1.6 polls `agent-logs.jsonl` every 5s; v1.7 ties this to the cloud trace stream for Pro+ users.
- Cross-machine views — v1.6 shows the local machine's automations only.

## Data sources

| Source | Where it comes from | Node shape |
|---|---|---|
| **Sequential groups** | `agent_groups` table where `dispatch_kind = 'sequential'`, joined with `agent_group_members` | One node per child agent, edges `position N → position N+1`, group title as a containing frame |
| **Routed groups** | `agent_groups` table where `dispatch_kind = 'routed'`, plus `router_config` JSON | Router node in center, child agents as fan-out, each `routerRules[i]` becomes a labeled edge |
| **Cron jobs** | `~/.ato/cron-jobs.json` (read via `list_cron_jobs`) | Cron node with `clock` icon at left edge, edge to the agent or group the cron dispatches to |
| **Skill flows** | Existing skill-detail parsing in `lib/skill-to-workflow.ts` | Same shape as today (preserve backward compat) |
| **Hooks** | `agent_hooks` table joined with `agents` | Small nodes attached to the input side of their parent agent |

Live status per node comes from a new helper `getRecentAgentRuns(agentSlug, limit=10)` reading `~/.ato/agent-logs.jsonl` — already used by Insights, so no new persistence.

## Critical files

### New
- `apps/desktop/src/lib/automations-aggregator.ts` — single function `getAllAutomations()` that returns `{ groups, crons, skills, hooks }` with normalized node + edge shapes the canvas can render.
- `apps/desktop/src/components/automation/sources/groupsToFlow.ts` — converts `AgentGroup` records to canvas nodes/edges.
- `apps/desktop/src/components/automation/sources/cronsToFlow.ts` — converts cron jobs to canvas nodes/edges.
- `apps/desktop/src/components/automation/SourceFilterChips.tsx` — the four toggle chips at the top of the canvas.

### Modified
- `apps/desktop/src/components/AutomationFlow.tsx` — replace the skill-only `useEffect` with a multi-source loader; keep skill-flow rendering as one of the four sources.
- `apps/desktop/src/stores/useAutomationStore.ts` — extend the workflow shape with a `source: "group" | "cron" | "skill" | "hook"` field so the filter chips can hide/show.
- `apps/desktop/src/components/automation/constants.ts` — add color tokens + icons for the four source types.

### Reused (no changes)
- `lib/skill-to-workflow.ts` — kept as the skill-flow source.
- `~/.ato/agent-logs.jsonl` — read for live status (same file Insights uses).
- The canvas rendering layer (`FlowCanvas.tsx`, `NodePalette.tsx`, `ExecutionOverlay.tsx`) is source-agnostic — no edits needed.

## Build order

**Wave 1 — sources (week 1)**
1. `automations-aggregator.ts` skeleton — return empty arrays for each source.
2. `groupsToFlow.ts` — read agent_groups + members, render sequential as left-right and routed as fan-out.
3. Wire the aggregator into `AutomationFlow.tsx` behind a feature flag; keep the existing skill source as fallback.

**Wave 2 — cron + hooks (week 2)**
4. `cronsToFlow.ts` — read cron jobs, attach clock-icon head node to dispatch target.
5. Hook source — read `agent_hooks` and attach as input nodes to their agent.

**Wave 3 — live status + UX (week 3)**
6. `getRecentAgentRuns()` — single-pass scan of `agent-logs.jsonl`, build agent-slug → last-status map.
7. Status pill on each node + click-through into Insights.
8. Source filter chips at the top of the canvas.
9. Empty state copy.

**Wave 4 — polish (week 4)**
10. Routed group rule labels on edges (e.g., "if perf, slow, N+1").
11. Cron next-run timestamp in node header.
12. Layout pass — force-directed for routed groups, left-to-right for sequential, vertical stacking for crons.
13. Docs + V1.6.0-IMPLEMENTATION.md + memory.

Total: ~4 weeks for v1.6 alongside the other Intelligence Layer items.

## Verification checklist

When v1.6.0 ships:

1. Create a sequential group `write-and-review` (Claude writer → Codex reviewer). Open Automations tab. Confirm the flow renders left-to-right with the right runtime colors and a "stage 1/2" pill on each child.
2. Create a routed group `code-review-team` (perf-reviewer + code-reviewer + a router rule). Confirm the router node sits in the center and the rule keywords ("performance, slow, N+1") render on the edge to perf-reviewer.
3. Create a cron job dispatching `@security-reviewer` weekdays at 9am. Confirm a clock-icon node appears on the left with `Mon-Fri 09:00` and an edge to `@security-reviewer`.
4. Run the sequential group once via the chat. Refresh the tab; both child nodes show "succeeded" + last-run timestamp.
5. Force one stage to fail (e.g., revoke Codex creds). Refresh; the failed node shows red status and the edge after it stops at the failure point.
6. Click a node → Insights opens with that agent's trace explorer pre-filtered.
7. Toggle "Groups" off in the filter chips → only crons + skills remain visible.
8. Existing skill flows still render under the "Skills" filter (no regression for users who relied on the v1.5 behavior).
9. Empty state shows when nothing is configured, with copy pointing at the four ways to create automations.
10. ROADMAP, PRD, V1.6.0-IMPLEMENTATION.md, CLAUDE.md updated.

## Tier gating

Free users see all their local automations on the canvas — same data the rest of the app shows. The Pro/Team tier value remains:
- **Cloud trace retention** for click-through into runs older than the local 100-line buffer (Pro)
- **Team-aggregate views** showing automations across teammates (Team)
- **SLA dashboards** showing automation health by SLA target (Enterprise)

The canvas itself is Free.
