# v1.6.0 — Automations Tab Repurpose

## What shipped

Per [V1.6.0-AUTOMATIONS-REPURPOSE.md](V1.6.0-AUTOMATIONS-REPURPOSE.md), the Runs → Automations tab is no longer just a skill-flow viewer. It now renders **everything that runs without a human in the loop** on one canvas:

| Source | What you see | Wave |
|---|---|---|
| **Skill flows** (existing) | One workflow per skill with `## Step N` / `## Phase N` headers, parsed from SKILL.md | pre-existing |
| **Sequential agent groups** | Left-to-right pipeline; each child carries its runtime so cross-runtime chains (Claude → Codex → Gemini) read clearly | Wave 1 |
| **Routed agent groups** | Router node in the center, child agents fan out, edge labels show the matching keyword rule (first 3 keywords) | Wave 1 |
| **Cron jobs** | Clock-trigger node with the human-readable schedule and next-fire timestamp; edge → dispatch target (agent / group / runtime) | Wave 2 |
| **Pre-call hooks** | Hook nodes stacked on the input side, edges labeled `<context>`, feeding into their parent agent on the right | Wave 2 |
| **Live status** | Every agent-attributed node shows idle / active / error pulled from `agent-logs.jsonl` metrics, refetched every 30s | Wave 3 |

Source filter chips at the top of the canvas auto-show the source types present in the current workspace (Skills / Pipelines / Routed Groups / Schedules / Hooks). Click a chip to scope the workflow list.

## Critical files

### New

- `apps/desktop/src/lib/automationsAggregator.ts` — single module with five exports:
  - `groupToWorkflow(group)` — convert one `AgentGroup` to a `Workflow`. Branches on `dispatchKind` for sequential vs routed layout.
  - `groupsToWorkflows(groups)` — bulk version.
  - `cronToWorkflow(cron, agents, groups)` — clock-trigger flow rooted at the schedule, pointing at the dispatch target.
  - `cronsToWorkflows(crons, agents, groups)` — bulk version.
  - `hooksToWorkflow(agent, hooks)` — agent-rooted flow with hook nodes feeding `<context>` edges into the agent.
  - `hooksToWorkflows(agents, hooksByAgentId)` — bulk version (skips agents with no hooks).
  - `decorateWorkflowsWithStatus(workflows, perAgent)` — overlays live status from `PerAgentMetrics`. Pure function; returns new workflow objects rather than mutating.

### Modified

- `apps/desktop/src/components/AutomationFlow.tsx` — new `useQuery`s for `listAgentGroups`, `listAgents`, `listCronJobs`, fan-out `listAgentHooks`, and `getAgentMetrics`. A single effect merges all four source types into the automation store and decorates them with live status.
- `apps/desktop/src/components/automation/types.ts` — `WorkflowSource` extended with `group-routed`, `group-sequential`, `hook` so the existing source filter automatically picks them up.
- `apps/desktop/src/components/automation/WorkflowToolbar.tsx` — friendly labels for the new sources in the filter chip strip.

### Reused (no changes)

- The existing `FlowCanvas` rendering layer is source-agnostic — adding new sources cost zero canvas changes.
- `lib/skill-to-workflow.ts` is still the skill source; it sits beside the new aggregator rather than being swallowed by it.
- `~/.ato/agent-logs.jsonl` (read via `getAgentMetrics()`) is the same data Insights uses — the canvas reuses that pipeline.

## What didn't ship in v1.6.0 (deferred)

Per the original plan but explicitly out-of-scope for this release:

- **Drag-and-drop authoring of new groups from the canvas.** Groups are still authored in `Agents → Groups → New`. Coming in v1.7+.
- **Click-through from a node into Insights.** Currently nodes select for a detail panel; deep-linking into the per-agent trace explorer is queued for v1.7.
- **Real-time WebSocket status.** v1.6 polls `agent-logs.jsonl` every 30s; v1.7+ ties this to the cloud trace stream for Pro users.
- **Cross-machine views.** v1.6 shows the local machine's automations only.

## Verification

1. Create a sequential group (e.g., `write-and-review`: code-writer Claude → security-reviewer Codex). Open Runs → Automations, switch the workflow to that group → confirm both nodes render with the right runtime tag, edges animate in left-to-right order.
2. Create a routed group with a keyword rule. Confirm the router node sits in the middle, children fan out, and the rule keywords show on the edge to the matching child.
3. Create a cron job dispatching `@security-reviewer` weekdays at 9am. Refresh the Automations tab → confirm the trigger node reads "Every weekday at 9am" with `Next: in N hours`, edge points to the agent.
4. Add a `file` pre-call hook to an agent reading a project file. Refresh → confirm a hook node appears on the input side of that agent's workflow with edge label `<context>`.
5. Run the same agent through chat a few times. Wait 30s. Refresh → confirm the agent node's status pill flips from idle to active (or error if the runs failed).
6. Toggle each chip in the filter strip → confirm the workflow list scopes accordingly.

## Tier gating

Free users see all their local automations on the canvas — same data the rest of the app shows. Pro/Team value:

- **Cloud trace retention** for click-through into runs older than the local 100-line buffer (Pro)
- **Team-aggregate views** showing automations across teammates (Team, future)
- **SLA dashboards** showing automation health by SLA target (Enterprise, future)

The canvas itself is Free.
