## §C. NEW — v2.17 Web ⇄ Desktop Tether

Branch: `v2.17/web-tethered-to-desktop` (folded into v2.18 branch).
Needs the **desktop app running** and an authenticated web session for
the same user. The tether channel currently supports one frame kind
(`decrypt_events`); v2.18 will add `dispatch_request`.

### C1. Pairing flow
- [ ] Open shared workspace page on web while desktop is running.
- [ ] `TetherStatusCard` initially shows "pending_approval" with a 12-char `browser_pubkey_fp` (formatted as 4-char groups, e.g. `a1b2 c3d4 e5f6`).
- [ ] Desktop raises a `TetherApprovalModal` showing the SAME `browser_pubkey_fp`.
- [ ] Verify the two fingerprints visually match before approving.
- [ ] Click **Approve once** → web card flips to "connected"; only this browser session uses the tether.
- [ ] Re-load the web page → tether re-enters "pending_approval" (no persistent grant).
- [ ] Click **Approve always** on a fresh pair → desktop writes a row to `tether_approvals`; subsequent re-pairs from the same browser+device auto-approve.

### C2. Defense-in-depth (#79 hardening)
- [ ] Manually edit `~/.ato/local.db` `tether_approvals` row, then re-pair. Cloud says auto-approved; desktop ALSO checks local DB before promoting → consistent state.
- [ ] In `~/.ato/ato.log` after a UA-hash mismatch, only the first 8 chars of the hash appear (no full-length identifier leak).

### C3. Heartbeat + sweeper
- [ ] On pair, `tether_hosts_online` row inserts in cloud DB (`POST /api/teams/tether/host/heartbeat`).
- [ ] Stop the desktop. Within the sweeper TTL window, the row is reaped and the web tether card flips to "host_offline".
- [ ] Restart desktop → heartbeat resumes → web flips back to "connected".

### C4. decrypt_events round-trip (the only Wave 1 frame kind)
- [ ] Open an E2E-shared resource (from §D below) on the web while tethered.
- [ ] The page sends a `decrypt_events` request through the tether; desktop reads `~/.ato/local.db` for the team key, decrypts, returns plaintext events.
- [ ] Events appear in the web page; no `__decrypt_error` banner.
- [ ] Turn off the desktop → web shows the connection indicator amber → red.

## §F. NEW — #70 War-room sweep CLI

> **Note**: lives on its own branch `feature/auto-close-war-rooms`
> (commit `f11124d`). To exercise, `git checkout
> feature/auto-close-war-rooms` and rebuild the CLI.

Purpose: replace the manual `ato war-rooms close <id>` step that's
been the #1 UX trap — fan out an R1 review, walk away, and have the
war-rooms self-close so they appear in Sessions feed.

### F1. CLI dry-run
- [ ] `ato war-rooms sweep --idle-minutes 30 --dry-run` returns ONE JSON envelope:
      `{ action: "sweep", dry_run: true, eligible: N, candidates: [...] }`.
- [ ] `--human` mode streams one line per candidate + a final summary.
- [ ] Candidates ordered oldest-idle first.

### F2. Validation
- [ ] `ato war-rooms sweep --max-per-run 0` → clap rejects ("0 is not in 1..").
- [ ] `ato war-rooms sweep --coordinator ""` → clap rejects ("value must not be empty").
- [ ] `ato war-rooms sweep --idle-minutes=-1` → clap rejects.

### F3. Real close
- [ ] Fire a war-room (e.g. claude + codex via `ato dispatch ... --war-room-id <uuid> --war-room-round 1`).
- [ ] Wait > idle-minutes.
- [ ] `ato war-rooms sweep --idle-minutes 5 --max-per-run 1 --coordinator google` → closes ONE war-room; emits ONE JSON envelope with closed_records.
- [ ] War-room appears in Sessions feed with auto-generated title / summary / tags.
- [ ] Re-run sweep → that war-room is excluded (already in `war_rooms` with status='closed').

### F4. Cron / launchd readiness
- [ ] Add a launchd `.plist` invoking `ato war-rooms sweep --idle-minutes 15` every 5 min.
- [ ] After 2 ticks, verify no double-close (each WR closes exactly once).

## §L. How to use this checklist

2. Walk §A (basic desktop tour) — anything broken is a hard blocker.
3. Walk §B and §C in browser + desktop side-by-side; these are the
   surfaces most likely to have regressed since v2.15.
4. §D is a v2.15 regression check, not new — should be quick.
5. §E + §F exercise the CLI changes (#70, #71, #78) end-to-end.
6. §G is informational — just confirm the v2.18 doc is sane.
7. §H requires PRO login. Hardest to validate without two accounts;
   prioritize H1 / H2 / H3 if time-boxed.
8. §I is a regression suite to re-run after the next code change.
9. Anything marked `[!]` becomes a follow-up task or a hotfix branch.

Happy hunting. Files / line refs welcome on regressions — they make
fix sweeps surgical instead of forensic.
