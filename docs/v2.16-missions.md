## What a Mission is

A **Mission** is a persisted goal-driven coordinator. Humans state:

- **A goal** in natural language ("ship the v2.17 onboarding flow refresh")
- **Success criteria** as verifiable receipts (tests pass, files changed, methodology score Ōēź X, owner approves)
- **Escalation policy** (when to ask the owner, when to abandon)
- **Workspace strategy** (single CWD or per-agent worktrees)
- **Merge strategy** (how the coordinator integrates parallel agent outputs)

The Mission then **runs over time, not in one shot.** It wakes on a tick, reads its own state + recent events, decides the next action (dispatch / escalate / mark complete / ignore), and persists the decision narrative.

The reactive shape (war-rooms + sessions) is preserved ŌĆö humans still convene war-rooms for decisions. Missions add the **proactive** shape: ATO drives work toward stated goals across days/weeks, with owner-facing triage.

## The shape ŌĆö three locked decisions

War-room verdicts from 2026-06-11. See `ROADMAP.md` v2.16 section for full design table.

### Q1: Mission relates to Loop via composition, not inheritance ŌĆö [VERDICT B-lite]

Missions are a **separate primitive** that spawns Loops as workers. Mission state lives in `missions`; execution receipts stay in `loop_runs`/`loop_run_steps` (already battle-tested in v2.14/v2.15). The relationship is recorded in `mission_events` payloads ŌĆö no join table.

| Why not A (Mission IS a Loop) | Why not C (Mission spawns Dispatches directly) |
|---|---|
| Loop is execution-oriented (already accreted `paused`+`skipped` states beyond original schema comment). Piling Mission's weeks-long lifecycle on top conflates two abstractions. | Throws away the only existing reusable execution substrate (graph topology, step audit, methodology integration, pause/resume). Mission would have to reimplement all of it. |

### Q2: Workspace boundary is a declarative field, executed imperatively ŌĆö [VERDICT W2]

Mission has a `workspace_strategy` field. Default `single_cwd`. The `per_agent_worktree` value triggers coordinator code that creates git worktrees from a base SHA, dispatches each agent into its own worktree, and tracks the worktree path in `paused_dispatches.workspace_root` (the seam already exists from v2.15.4).

| Why not W1 (shared CWD always) | Why not W3 (separate `mission_workspaces` table) |
|---|---|
| ATO already treats same-workspace overlap as attribution ambiguity (`apps/desktop/src-tauri/src/active_runs.rs:89-93`). Shared-CWD with parallel agents is the exception, not the foundation, for a Mission that explicitly coordinates multiple coding agents. | Premature subsystem. `paused_dispatches.workspace_root` is reusable. Don't add a top-level workspace primitive until there's a use case beyond Mission. |

### Q3: Merge strategy is a declarative field; execution is coordinator code ŌĆö [VERDICT M2]

Mission has a `merge_strategy` field. Values:

- `human_approves_each` ŌĆö coordinator presents diffs, human approves each agent's contribution individually
- `coordinator_merges_all` ŌĆö coordinator dispatches a merge step that integrates all worktrees into a single result
- `coordinator_picks_winner` ŌĆö coordinator selects the single best output (ranked by `success_criteria`)
- `ranked_by_score` ŌĆö agents are ranked, top-N's contributions merged

| Why not M1 (no primitive, human merges manually) | Why not M3 (merge is a typed Loop node) |
|---|---|
| Collapses Mission back into a fan-out launcher. The whole point of a proactive coordinator is that it integrates results. | Most non-dispatch Loop kinds are still stubbed (`apps/cli/src/commands/loops.rs:1007-1023`). Forcing merge to be a typed Loop node couples Mission rollout to unfinished Loop graph semantics. |

### Cross-question consistency rule

If `workspace_strategy = per_agent_worktree`, then **the merge executor must consume explicit per-agent `workspace_root` + base SHA**. Otherwise provenance is lost and merges can't be deterministically audited.

### Concurrency safety rules (gemini round 3)

- **`single_cwd` + parallel agents is unsafe by default.** Two agents running `cargo build` (or any build that touches `target/`) in the same directory will race. PR-1.5 (API-provider tool surface) enforces a soft check: when `workspace_strategy=single_cwd`, the coordinator tick refuses to spawn more than one in-flight Dispatch under a Mission at a time. Override is opt-in (`--allow-concurrent-single-cwd`).
- **CLI worktree-targeting (PR-3+):** when `per_agent_worktree`, every dispatch under a Mission MUST carry an explicit `workspace_root` so the CLI/API tools run their tool calls (read_file, edit_file, bash) inside the right worktree, not the root. The seam exists today (`paused_dispatches.workspace_root` from v2.15.4) ŌĆö extend it to `dispatch::run()` signature.
- **`mission_id` is nullable on related rows.** Standalone loops (the v2.14 use case) keep working without a Mission parent. Mission Ōåö Loop relationship is established via `mission_events.payload.loop_run_id` when a Mission spawns a Loop, NOT via a `loop_runs.mission_id` FK column that would force every loop to belong to a Mission.

## Other locked design decisions (from Steinberger teardown)

- **4 triage categories:** `autonomous` / `needs_owner` / `ignored` / `done`. The `ignored` category is a Steinberger borrow ŌĆö items the owner has explicitly told the Mission to skip stay visible (never silently closed).
- **No-subdelegation rule:** Missions spawn Loops (and Loops spawn Dispatches). Dispatches CANNOT spawn Missions. Loops CANNOT spawn sub-Loops. Single root orchestrator.
- **Decision briefs on escalation:** When a Mission asks the owner for a decision, it emits a brief ŌĆö tradeoffs, proof state, recommendation, exact choices ŌĆö not a URL. Borrowed verbatim from the v2.15.4 abandon path; same shape.
- **Live proof before landing:** External-integration steps require authenticated live calls. Mocks supplement live proof; they don't replace it.
- **Persistent narrative:** Each Mission gets a `~/.ato/missions/<slug>.md` running narrative alongside the SQLite row. Owner-inspectable without booting the desktop. The narrative explains WHY actions were taken; it is NOT the authoritative plan.
- **Pre-flight on resume:** When a Mission tick wakes, re-verify pre-conditions (runtimes still authed, git state clean, base SHA still exists) before re-firing.

## Schema (PR-1 surface)

Two new tables. Drop into `apps/desktop/src-tauri/src/schema.rs` next to `loops`/`loop_runs`.

> **`merge_strategy` clarification (gemini round 3 misread):** these values describe the **coordinator's integration strategy** ŌĆö how it decides which agent's work to accept ŌĆö NOT git merge mechanics. The actual git operations (cherry-pick, three-way merge, conflict resolution) happen *inside* the coordinator code regardless of which strategy is selected. Don't conflate this field with `git merge --strategy=...`.

> **`category` is operational policy, not metadata (gemini round 3 pushback rejected):** the coordinator tick reads `category` to decide behavior. `ignored` means "owner said skip, don't escalate again." `needs_owner` means "block on owner approval at each escalation." `autonomous` means "act without owner approval." `done` is the terminal state. Removing this field collapses the Steinberger triage primitive ŌĆö the entire reason for the field's existence.

```sql
CREATE TABLE IF NOT EXISTS missions (
    id                  TEXT PRIMARY KEY,
    slug                TEXT NOT NULL UNIQUE,
    name                TEXT NOT NULL,
    goal                TEXT NOT NULL,

    -- v2.16 gemini-round-3: success_criteria carries both LLM-readable
    -- text AND a programmatic check_command so "complete" is verifiable,
    -- not subjective.
    success_criteria    TEXT NOT NULL,        -- JSON: [{kind, description, check_command, met_at}]
                                              -- check_command runs in workspace_root with exit-0 = met

    escalation_policy   TEXT,                 -- JSON: {ask_owner_when, abandon_when}
    workspace_strategy  TEXT NOT NULL DEFAULT 'single_cwd',
                                              -- 'single_cwd' | 'per_agent_worktree'
    base_sha            TEXT,                 -- for per_agent_worktree, the SHA worktrees branch from

    -- v2.16 gemini-round-3 biter: prevents ~/.ato/missions/<slug>/worktrees/
    -- from filling up disk and cluttering `git worktree list`.
    cleanup_policy      TEXT NOT NULL DEFAULT 'delete_on_success',
                                              -- 'retain' | 'delete_on_success' | 'always_delete'

    merge_strategy      TEXT NOT NULL DEFAULT 'human_approves_each',
                                              -- coordinator integration strategy, NOT git mechanics:
                                              -- 'human_approves_each' | 'coordinator_merges_all'
                                              -- | 'coordinator_picks_winner' | 'ranked_by_score'
    category            TEXT NOT NULL DEFAULT 'autonomous',
                                              -- 'autonomous' | 'needs_owner' | 'ignored' | 'done'
    state               TEXT NOT NULL DEFAULT 'open',
                                              -- 'open' | 'in_progress' | 'blocked' | 'complete'

    -- v2.16 gemini-round-3: bounded resource consumption per Mission so
    -- the coordinator tick can't burn $$$ in a stuck in_progress loop.
    max_loops           INTEGER,              -- NULL = no cap; coordinator refuses to spawn beyond this
    token_budget_usd    REAL,                 -- NULL = no cap; sum of execution_logs.cost_usd against this

    -- v2.16 gemini-round-3: declarative outputs ŌĆö PR URL, artifact paths,
    -- generated reports. Coordinator writes here on completion.
    result_metadata     TEXT,                 -- JSON: { pr_url, artifact_paths[], summary, ... }

    narrative_md_path   TEXT NOT NULL,        -- ~/.ato/missions/<slug>.md
    created_at          TEXT NOT NULL,
    updated_at          TEXT NOT NULL
);

CREATE INDEX idx_missions_state_category ON missions(state, category);
CREATE INDEX idx_missions_updated ON missions(updated_at DESC);

CREATE TABLE IF NOT EXISTS mission_events (
    id                  TEXT PRIMARY KEY,
    mission_id          TEXT NOT NULL REFERENCES missions(id) ON DELETE CASCADE,
    kind                TEXT NOT NULL,        -- 'planned' | 'dispatched' | 'loop_run_completed'
                                              -- | 'escalated' | 'owner_decision' | 'success_check'
                                              -- | 'category_changed' | 'state_changed' | 'completed'
                                              -- | 'abandoned'
    payload             TEXT,                 -- JSON, kind-specific (may include loop_run_id,
                                              -- paused_dispatch_id, dispatch_id, owner_text, etc.)
    occurred_at         TEXT NOT NULL
);

CREATE INDEX idx_mission_events_mission_time ON mission_events(mission_id, occurred_at DESC);
CREATE INDEX idx_mission_events_kind ON mission_events(kind, occurred_at DESC);
```

**No `mission_loops` join table** ŌĆö the Mission Ōåö Loop relationship lives in `mission_events.payload` as `loop_run_id` when `kind='loop_run_completed'`. Codex's B-lite refinement. Single source of truth for state evolution.

## Why this matters ŌĆö the Fable wedge

Per the [[sully-fable-thesis]] anchor: with Fable-class models no longer subsidized, post-subsidy AI bills 5x. The unfair advantage of v2.16 Missions over Steinberger's open-sourced codex skills (`maintainer-orchestrator`, `github-project-triage`):

> Steinberger's skills **explicitly omit model selection**: *"Do not set or request a custom model; omit model selection and inherit the platform default."*

ATO's Mission can do what they punted on: **for each step in a Mission, pick the cheapest model that passes the rubric** (the v2.10 methodology runner already does this), and **show receipts** (`execution_logs.cost_usd`, `dispatches show <id> --human`). In a 5x-cost world, that's the dashboard, not a curiosity.

**Positioning lead-line for v2.16:** NOT "scheduled loops" (Steinberger has those). The line is **"loops that pick the right model per step, with receipts, and merge parallel agent work as a team."**

## PR-4 design ŌĆö coordinator tick (war-room `D6A23631-0F0C-4E7E-8B26-2A361B825DDE`, locked 2026-06-12)

Two cross-family seats (codex + gemini), unanimous on 4 of 5:

| Q | Verdict | Shape |
|---|---|---|
| Q1 trigger | **A** (unanimous) | One-shot `ato missions tick` run by a scheduled job (launchd/cron, mirroring the v2.15.4 startup scanner). No daemon ŌĆö avoids liveness/PID-ownership/crash-recovery problems the architecture has deliberately avoided. |
| Q2 actions per wake | **A** (unanimous) | Exactly one action per mission per tick. "If cadence is too slow, fix cadence, not semantics" (codex). |
| Q3 criteria cadence | **B + escape hatch** (synthesis: codex B vs gemini A) | check_commands run only when terminal worker events landed since the last `success_check` (check_commands can be whole test suites ŌĆö polling is expensive). Gemini's ground-truth concern (owner resolved things manually) is served by `ato missions check <slug>`, which forces evaluation on demand. |
| Q4 non-blocking spawn | **B** (unanimous) | Detached child process + new pre-completion `dispatch_started` event {attempt_id, pid}; `dispatched` closes it by attempt_id. Fixes the blind spot: `dispatched` is only written after a synchronous dispatch completes. Stale starts (dead pid, no close) get `dispatch_abandoned`. |
| Q5 failure guard | **C** (unanimous) | Budgets AND 3-consecutive-failures ŌåÆ `blocked` + decision brief. Codex: budgets only count *persisted* work, so pre-persist failures could retry forever without it. |

Implementation additions (flagged for review, not war-roomed): nullable `worker_config` JSON column ({runtime, model, require_tools}) tells the tick *what* to dispatch; unset ŌåÆ single deduped `escalated` event instead of action. Escalations dedup by payload.reason until an owner_decision/state_changed supersedes. No-op ticks write zero events (96 wakes/day must not pollute the audit log). Smart per-step worker planning (cheapest-model-per-step, the Fable wedge) is deliberately NOT in PR-4 ŌĆö it composes with the v2.10 methodology runner in a later slice.

## Open questions for v2.16 PR-5+

- How does `coordinator_picks_winner` rank agents without a judge LLM? Heuristic options: most files changed that pass tests; smallest diff that meets success_criteria; the agent whose own `success_criteria_met` field returned true first.
- Which workspace_root do success_criteria check_commands run in for `per_agent_worktree` missions with multiple agents? PR-4 runs them in `repo_root`; per-worktree evaluation is PR-5 (merge) scope.
- How does `per_agent_worktree` interact with the desktop's CWD-based project picker? The Mission's worktrees live under `~/.ato/missions/<slug>/worktrees/<agent>/`.

These are PR-2+ design slice topics ŌĆö not blockers for PR-1 schema.

## API-providers-as-coding-agents (load-bearing dependency)

**Driver clarification 2026-06-11:** Missions coordinate *multiple coding agents working in different worktrees, all merged as a team by the coordinator.* Today only the **CLI runtimes** (claude / codex / gemini CLI binaries) can edit/write code; **API providers** (anthropic, openai, google, minimax) only have `read_file` + `grep` via `apps/cli/src/api_dispatch_tools.rs::dispatch_with_tools()`. They can review, but they can't be coding-agent workers.

This is a v2.16-blocker: with only CLI-runtime coding agents, Missions are limited to a 3-runtime team and can't pick the cheapest model per step for non-CLI providers (the Fable-wedge value). API providers must get a write-tool surface BEFORE the Mission tick lands.

| Tool | Today (read-only loop) | Need for coding agents |
|---|---|---|
| `read_file` | Ō£ģ | Ō£ģ |
| `grep` | Ō£ģ | Ō£ģ |
| `list_dir` | partial | Ō£ģ |
| `edit_file` (string replace with old_string/new_string) | ŌØī | Ō£ģ |
| `write_file` (full file create/overwrite) | ŌØī | Ō£ģ |
| `bash` (constrained: workspace_root scoping + allowlist) | ŌØī | Ō£ģ for build/test verification |
| `git_status` / `git_diff` (read-only git ops) | ŌØī | Ō£ģ for merge-strategy provenance |

**Open-core split for this surface:** all 7 tools ship OSS. The PRO upgrade is hosted policy enforcement (which tools any agent may call, per-runtime), cross-machine audit aggregation, and team-level allowlists. Local enforcement (workspace_root scoping, deny `rm -rf /`, deny network in `bash`) is OSS.

**Open-core boundary on the bash tool specifically:** OSS ships with a conservative allowlist ŌĆö `cargo`, `npm`, `pytest`, `git`, common build/test commands. The "expanded allowlist with full hosted policy enforcement" ships PRO. This matches Steinberger's authorization-granularity pattern (triage ŌēĀ edit ŌēĀ push ŌēĀ merge ŌĆö each is a separate permission).
